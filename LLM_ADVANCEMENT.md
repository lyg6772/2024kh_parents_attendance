# LLM 고도화 분석 및 방안 (LLM Advancement)

본 문서는 학부모 보람교사 출석 관리 시스템의 기존 SSR(Server-Side Rendering) 방식과 새롭게 도입된 LLM 에이전트 엔진의 기술적 차이를 분석하고, 향후 시스템 고도화를 위한 실무적 가이드를 제시한다.

## Part 1: 기존 SSR 개발 vs LLM 엔진 비교

### 1.1 요청 처리 흐름 비교
- **SSR (결정론적 경로)**:
    - **흐름**: 사용자 UI 조작(캘린더 클릭) → `POST /admin/attendee` → `AdminAttendeeService` → `functions.py(DAO)` → DB.
    - **특징**: 입력과 출력이 1:1로 매핑된다. 사용자는 정해진 버튼을 눌러야만 기능을 실행할 수 있다.
- **LLM (비결정론적 ReAct 루프)**:
    - **흐름**: 사용자 자연어 입력 → `engine.run()` → LLM이 상황 판단 및 도구 선택 → `attendance_data.py` 호출 → DB → 결과 요약.
    - **실제 사례**: "오늘 참석자 김철수 추가해줘" 요청 시
        1. `get_attendance(yyyymm)` 호출하여 현재 상태 확인.
        2. `save_attendance(date, attendee)`를 조합하여 변경 사항 생성.
        3. 사용자에게 최종 승인을 요청하는 `pending_confirmation` 상태로 전환.
    - **차이**: SSR이 단일 기능 실행에 그친다면, LLM은 상태를 확인하고 필요한 단계를 스스로 조합(Multi-step)한다.

### 1.2 기능 추가 방식 비교
- **SSR 방식**:
    - 새 기능을 추가하려면 라우터 등록, 컨트롤러 로직 작성, Jinja2 템플릿 수정, 프론트엔드 JS 이벤트 리스너 추가가 모두 필요하다.
- **LLM 엔진 방식**:
    - `tools.py`에 `ToolDefinition`을 추가하고 핸들러를 등록하면 끝난다. `prompts.py`의 `build_system_prompt()`가 레지스트리를 읽어 LLM에게 자동으로 기능을 가르친다.
    - **예시**: `navigate_month` 도구 추가 시 SSR은 별도의 URL 처리와 이동 로직을 UI에 심어야 하지만, LLM은 "5월 화면 보여줘"라는 말 한마디로 `redirect` 응답을 생성하여 클라이언트를 제어한다.

### 1.3 에러 처리 및 가드레일
- **SSR**: 데이터 형식이 맞지 않으면 400 에러를 뱉거나 로직 내 `try-except`로 예외를 관리한다. 예측 범위가 좁다.
- **LLM**: Hallucination(환각)이나 잘못된 인자 추출 위험이 있다.
    - **대응**: 본 프로젝트는 Pydantic을 통한 도구 인자 검증, `MAX_TURNS=5`를 통한 무한 루프 방지, 쓰기(WRITE) 작업 전 사용자 명시적 승인(`confirm`) 절차를 두어 비결정론적 위험을 제어한다.

### 1.4 사용자 경험의 공존
- **SSR**: 학습 곡선이 존재하지만 응답이 즉각적이고 예측 가능하다. 대량의 데이터를 표 형식으로 볼 때 유리하다.
- **LLM**: 자연어로 복합적인 의도를 처리(예: "지난달이랑 이번달 출석 다 엑셀로 뽑아줘")할 수 있으나, API 호출 지연(Latency)이 존재한다.
- **결론**: 관리자 페이지(`admin_attendee.html`) 내에 플로팅 채팅 패널로 공존시켜, 정교한 조작은 UI로, 복합 작업은 대화로 처리하도록 설계했다.

## Part 2: 고도화 방안

### 2.1 대화 히스토리 영속화
- **현황**: 프론트엔드 `chatHistory` 배열에 저장되어 페이지 새로고침 시 초기화됨.
- **문제**: 이전 맥락을 유지한 대화가 불가능함.
- **개선**: `KyChatLogL` 테이블 신설. 세션 ID별로 대화 내용을 DB에 저장하고, 엔진 시작 시 최근 5~10개의 메시지를 `messages` 배열에 주입.
- **난이도/우선순위**: 중간 / **최우선**.

### 2.2 LLM 요청 한도(Rate Limit) 관리
- **현황**: `FailoverAdapter`가 Groq 실패 시 Gemini로 전환하나, 둘 다 소진 시 일반 에러 발생.
- **문제**: Groq(30 RPM), Gemini(10 RPM)의 낮은 무료 티어 한도로 인해 다수 사용자 접속 시 취약함.
- **개선**: Redis 또는 메모리 내 사용자별 카운터 도입. 한도 초과 시 "현재 요청이 많습니다. N분 후 시도해주세요"와 같은 구체적 안내 제공. 캐싱 가능한 읽기 도구(`get_attendance`) 결과 저장.
- **난이도/우선순위**: 중간 / 높음.

### 2.3 도구 관리 확장성 (Tool Governance)
- **현황**: 도구 추가 시 `ToolDefinition`, 핸들러, 레지스트리 3곳을 수동 동기화해야 함 (TODO-001).
- **문제**: 도구가 10개 이상 늘어나면 시스템 프롬프트가 길어져 토큰 비용이 늘고 LLM의 판단력이 흐려짐.
- **개선**: 데코레이터(`@tool`) 기반 자동 등록 시스템 구축. 도구 카테고리화(Read/Write/Navigation)를 통해 1차적으로 필요한 도구 세트만 프롬프트에 제공하는 라우팅 도입.
- **난이도/우선순위**: 높음 / 중간.

### 2.4 관측성 (Observability)
- **현황**: 상세 로그가 부족하고 토큰 사용량 추적이 안 됨.
- **문제**: 에이전트가 왜 실패했는지, 비용이 얼마나 발생하는지 파악 불가.
- **개선**: `EngineResult`에 `usage` 필드 추가. 모든 `tool_call`과 응답 시간을 `agent_execution_log`에 기록. ELK 또는 단순 파일 로그로 구조화된 데이터 축적.
- **난이도/우선순위**: 중간 / 높음.

### 2.5 프롬프트 버저닝 및 최적화
- **현황**: `prompts.py`에 문자열로 하드코딩됨.
- **문제**: 지시문 수정 시마다 서버를 재배포해야 함.
- **개선**: YAML 또는 외부 프롬프트 관리 툴(LangSmith 등) 연동 검토. 현실적으로는 프로젝트 내 `prompts/` 폴더에 버전별 `.txt` 파일을 두고 로드하는 방식 추천.
- **난이도/우선순위**: 낮음 / 낮음.

### 2.6 보안 고도화
- **현황**: JWT 인증 및 쓰기 승인 절차 존재.
- **문제**: 프롬프트 인젝션(Prompt Injection)을 통해 시스템 정보를 캐내거나 권한 밖의 도구 실행 시도 가능성.
- **개선**: 입력 텍스트 길이 제한(최대 500자). 응답 필터링(PII 탐지). 도구 실행 전 `KyUserL`의 권한(Role)을 핸들러 내부에서 재검증.
- **난이도/우선순위**: 높음 / 중간.

### 2.7 멀티스텝 및 병렬 호출
- **현황**: `MAX_TURNS=5`, 1턴에 1개 도구만 호출 가능 (Groq/Gemini 제약).
- **문제**: "3월부터 5월까지 출석 내역 다 보여줘" 같은 요청 시 3번의 루프를 돌아야 하므로 느림.
- **개선**: 여러 날짜/월을 인자로 받는 '배치 처리용 도구' 추가. LLM이 병렬 도구 호출(Parallel tool calls)을 지원하도록 프롬프트 구조 개선.
- **난이도/우선순위**: 중간 / 중간.

### 2.8 테스트 전략 고도화
- **현황**: 유닛 테스트는 완비되었으나 LLM 응답 품질은 Mocking 처리됨.
- **문제**: 모델 변경이나 프롬프트 수정 시 실제 성능 하락을 감지할 수 없음.
- **개선**: '골든 데이터셋(질문-기대도구-기대인자)' 구축. 실제 API를 호출하는 통합 테스트를 정기적으로 수행하여 성공률(Pass Rate) 측정.
- **난이도/우선순위**: 높음 / 중간.

## Part 3: 개발 방법론의 전환 — 그리고 자체 LLM + RAG의 가능성

### 3.1 무엇이 달라지는가

기존 개발과 LLM 도구 엔진 개발은 개발자의 역할 자체가 다르다.

| | 기존 개발 | LLM 도구 엔진 |
|---|---|---|
| 개발자 역할 | **라우터** — 어떤 입력에 어떤 코드를 실행할지 흐름을 직접 작성 | **도구 제공자** — 시스템이 무엇을 할 수 있는지 선언, 조합은 LLM이 판단 |
| 코드 성격 | 명령형 — if/else, route matching, 이벤트 바인딩 | 선언형 — ToolDefinition + 핸들러 |
| 기능 추가 | 라우트 + 컨트롤러 + 서비스 + 템플릿 + JS 이벤트 | 도구 정의 + 핸들러 (UI 수정 불필요) |
| 테스트 대상 | "코드가 의도대로 동작하는가" | "LLM이 올바른 도구를 고르는가" + "도구가 의도대로 동작하는가" |

**핵심 차이**: 기존에는 개발자가 `사용자 의도 → 실행 코드`의 매핑을 모두 작성했다. LLM 엔진에서는 그 매핑을 LLM이 런타임에 수행하고, 개발자는 실행 가능한 능력(도구)만 정의한다.

이 프로젝트의 `navigate_month` 도구가 대표적이다. SSR이면 버튼 + JS 이벤트 + 라우터가 필요하지만, LLM 엔진에서는 4줄짜리 핸들러 하나로 "5월로 이동해줘", "다음 달 보여줘", "이전 달 출석부" 등 표현 방식에 관계없이 전부 처리된다.

### 3.2 이것이 새로운 방법론인가

**아직은 "방법론"이라기보다 "레이어 추가"에 가깝다.**

- 현재는 기존 SSR 위에 LLM 레이어를 얹은 구조다. 핵심 비즈니스 로직(DAO, 서비스)은 그대로이고, LLM은 그 위에서 "어떤 기능을 어떤 순서로 호출할지" 판단하는 역할만 한다.
- 이 구조에서 LLM이 죽어도 기존 UI로 업무가 돌아간다. 이것이 "방법론 교체"가 아닌 이유다.

**다만, 방향은 분명하다.** 기능이 늘어날수록 "UI를 새로 만드는 것"보다 "도구를 하나 등록하는 것"이 압도적으로 빠르다. 장기적으로 새 기능의 1차 인터페이스가 LLM이 되고, UI는 자주 쓰는 기능의 단축키 역할로 바뀔 수 있다.

### 3.3 현재 구조의 한계 — 왜 외부 API만으로는 부족한가

이 프로젝트가 Groq/Gemini 무료 API에 의존하면서 겪는 근본적 한계:

| 한계 | 원인 |
|---|---|
| **비결정론** | 범용 LLM은 우리 도메인을 모른다. "참석자 추가"와 "특이사항 입력"을 구분하는 건 프롬프트 튜닝에 의존 |
| **보안 불안** | 사용자 입력과 시스템 프롬프트가 외부 서버로 전송된다. 민감한 비즈니스 데이터가 3자에게 노출 |
| **Rate Limit** | 무료 티어(Groq 30 RPM, Gemini 10 RPM)는 사용자 10명만 동시 접속해도 소진 |
| **도구 폭발** | 범용 LLM에게 도구 50개를 주면 선택 정확도가 급격히 떨어짐 |
| **레이턴시** | 외부 API 왕복 1~3초/턴. 멀티스텝이면 5~15초 |

### 3.4 자체 LLM + RAG — 무엇이 달라지는가

자체 LLM 서버를 구축하고, RAG(Retrieval-Augmented Generation)로 도메인 지식을 주입하면 위 한계 대부분이 해소된다.

#### 3.4.1 도메인 특화로 비결정론 문제 완화

현재 `prompts.py`에서 시스템 프롬프트로 도구 설명을 전달하는 방식은, LLM이 "출석 관리"라는 도메인을 매 요청마다 프롬프트에서 학습해야 한다.

자체 LLM + RAG 구조에서는:

```
[사용자 입력: "김철수 오늘 출석 처리해줘"]
    ↓
[RAG 검색] → 관련 문서 retrieve
    - 출석 처리 플로우: get_attendance → save_attendance
    - 참석자 형식: 쉼표 구분 문자열
    - WRITE 작업은 반드시 사용자 확인 필요
    ↓
[LLM] → 정확한 도구 선택 + 올바른 인자 생성
```

RAG가 주입하는 것들:
- **제품 기획서** — "출석 등록 시 날짜는 YYYYMMDD, 참석자는 쉼표 구분"
- **비즈니스 규칙** — "WRITE 작업은 반드시 사전 조회 후 실행"
- **도구 사용 이력** — 과거에 비슷한 요청이 들어왔을 때 어떤 도구 조합이 성공했는지
- **도메인 용어집** — "보람교사 = 급식 도우미", "특이사항 = 해당 날짜의 메모"

프롬프트에 모든 걸 하드코딩하는 대신, 요청과 관련된 컨텍스트만 동적으로 검색해서 주입하므로 토큰 효율도 좋고 정확도도 높다.

#### 3.4.2 보안 문제 해소

| | 외부 API (현재) | 자체 LLM 서버 |
|---|---|---|
| 데이터 경로 | 사용자 입력 → 외부 서버(Groq/Google) → 응답 | 사용자 입력 → 내부 서버 → 응답 |
| 민감 데이터 | 3자 서버에 전송됨 | 내부망에서만 처리 |
| 프롬프트 인젝션 | 외부 서버가 보안 경계 밖 | 입출력 필터를 서버단에서 완전 제어 |
| 규제 대응 | 클라우드 제공자의 데이터 처리 약관에 종속 | 자체 인프라로 컴플라이언스 충족 |

학교 출석 데이터(학생/학부모 이름 등)는 개인정보에 해당할 수 있다. 자체 서버면 이 데이터가 외부로 나가지 않는다.

#### 3.4.3 도구 폭발 문제 해결 — 2단계 라우팅

도구가 50개 이상일 때, RAG를 활용한 2단계 도구 선택:

```
[1단계: 의도 분류 + 도구 검색]
사용자: "3월 출석 엑셀로 뽑아줘"
    ↓
RAG 검색 → 유사 요청 이력에서 "export_excel" 도구가 사용됨을 확인
    ↓
후보 도구 3~5개만 선별: [export_excel, get_attendance, navigate_month]

[2단계: 선별된 도구로 실행]
LLM에게 3개 도구만 제공 → 정확한 선택
```

범용 LLM에 50개 도구를 던지는 것과 달리, RAG가 관련 도구를 사전 필터링하므로 정확도가 유지된다.

#### 3.4.4 Rate Limit과 비용 문제

자체 서버면 요청 한도가 하드웨어 성능에만 의존한다. GPU 서버 1대로도 동시 접속자 수십 명을 처리할 수 있고, 토큰당 과금이 없으므로 사용량 증가에 따른 비용 증가가 선형적이지 않다.

다만 초기 구축 비용(GPU 서버, 모델 튜닝, RAG 파이프라인)이 발생하므로, 서비스 규모가 이를 정당화할 수 있는지가 판단 기준이 된다.

### 3.5 자체 LLM + RAG 구축 시 이 프로젝트에서 바뀌는 것

현재 아키텍처에서 변경이 필요한 지점:

```
현재:
  router.py → engine.py → FailoverAdapter(Groq/Gemini) → 외부 API
                         → build_system_prompt() (정적 프롬프트)

자체 LLM + RAG:
  router.py → engine.py → SelfHostedAdapter(vLLM/Ollama) → 내부 서버
                         → RAG 검색 → 동적 컨텍스트 주입
```

변경 범위:

| 컴포넌트 | 변경 | 비고 |
|---|---|---|
| `llm.py` | `SelfHostedAdapter` 추가 | `LLMAdapter` 인터페이스 덕분에 교체 용이 |
| `prompts.py` | RAG 검색 결과를 프롬프트에 동적 주입 | `build_system_prompt()`에 `context` 파라미터 추가 |
| `tools.py` | 변경 없음 | 도구 정의는 LLM 종류와 무관 |
| `engine.py` | 변경 없음 | ReAct 루프는 LLM 종류와 무관 |
| 인프라 | RAG 벡터 DB(Chroma/Milvus) + 문서 임베딩 파이프라인 추가 | 가장 큰 작업 |

`LLMAdapter` 추상 클래스 기반으로 설계한 것이 여기서 효과를 본다. `GroqAdapter`를 `SelfHostedAdapter`로 교체하면 나머지 코드는 그대로다.

### 3.6 현실적 판단 기준

| 규모 | 권장 | 이유 |
|---|---|---|
| 현재 (사용자 5~10명, 도구 5개) | 외부 API 유지 | 무료 티어로 충분. 자체 서버 구축 비용 대비 이득 없음 |
| 중규모 (사용자 50+, 도구 20+) | 유료 API 전환 + RAG 도입 | 무료 한도 초과. RAG로 도구 선택 정확도 보완 |
| 대규모 / 민감 데이터 | 자체 LLM + RAG | 보안 요구, 비용 최적화, 도메인 특화 모두 필요 |

### 3.7 결론

현재 이 프로젝트가 증명한 것: **"흐름 코딩"에서 "능력 선언"으로의 전환은 이미 작동한다.** 5개 도구, 무료 API만으로도 자연어 기반 출석 관리가 실용적으로 돌아간다.

자체 LLM + RAG는 이 전환을 "보조 레이어"에서 "핵심 인터페이스"로 격상시키는 열쇠다. 도메인 지식이 RAG에 축적될수록 LLM의 판단이 정확해지고, 결국 "이 시스템에 새 기능을 추가한다"는 것은 "기획 문서를 RAG에 넣고 도구 핸들러를 하나 작성한다"로 수렴한다. 그때가 되면 이것은 레이어 추가가 아니라 개발 방법론의 전환이라 할 수 있다.
